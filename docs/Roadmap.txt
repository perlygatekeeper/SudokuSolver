# SudokuSolver Development Roadmap

## Project Philosophy

SudokuSolver is a logic-first solver and analysis tool. Its primary goal is to
solve puzzles through techniques that a human can inspect and understand.

Development should continue to follow these rules:

* Prefer logical deduction over unrestricted brute-force search.
* Keep solving strategies independent and directly testable.
* Require every deduction to explain why it is valid.
* Preserve backward compatibility whenever practical.
* Favor readable, maintainable code over clever code.
* Keep commits small and focused.
* Use tests and stable benchmarks to prevent regressions.

Bounded hypothetical inference is acceptable when it produces a finite,
human-verifiable proof. Recursive guessing and completed speculative solves
should not silently become the basis for ordinary deductions.

## Completed Release Tracks

### Version 0.5.x — Project Revival and Infrastructure

Status: complete

* Restored the legacy repository and syntax checking.
* Added dependency tracking, versioning, a Makefile, and the initial test suite.
* Improved puzzle loading and command-line puzzle selection.
* Established developer documentation.

### Version 0.6.x — Strategy Architecture

Status: complete

* Introduced the Sudoku::Strategy namespace and strategy base class.
* Extracted legacy strategies into independent modules.
* Added structured Sudoku::Deduction objects.
* Added the strategy registry and deduction-driven solver orchestration.
* Preserved the tiered restart policy.
* Added strategy contracts and focused tests.

### Version 0.7.x — User-Facing Solver Features

Status: complete

* Added statistics, step-by-step solving, hint mode, and explain mode.
* Added versioned difficulty ratings.
* Added contradiction detection and benchmark support.
* Improved flexible puzzle-file handling.

### Versions 0.8.x–0.9.x — Advanced Logical Coverage

Status: complete

* Added triples and quads.
* Added Swordfish and Jellyfish through reusable fish infrastructure.
* Added XY-Wing, XYZ-Wing, and WXYZ-Wing.
* Added Unique Rectangle Types 1 through 4.
* Added Skyscraper, Two-String Kite, and Empty Rectangle.
* Added reusable strong-link infrastructure.
* Added Simple Coloring, Multi-Coloring, X-Chains, and XY-Chains.
* Added AIC, grouped inference nodes, Grouped L1-Wing, and Grouped AIC.
* Added bounded hypothetical propagation and Digit Forcing Chains.
* Expanded from the first-50 benchmark to the entire canonical corpus.

### Version 1.0.0 — Original Vision Complete

Status: complete and tagged

Canonical minimal 17-clue corpus:

```text
Puzzles processed : 49,158
Solved            : 49,158
Stalled           : 0
Contradictions    : 0
```

Version 1.0.0 also includes:

* Logic-first solving without unrestricted recursive backtracking.
* Structured and explainable deductions throughout the solver.
* Stable benchmark and regression infrastructure.
* Renderer abstraction and versioned event objects.
* Pretty, compact, candidate, Unicode, and machine-readable output formats.
* Candidate-state export and structured JSON solve results.
* A permanent three-digit test-numbering scheme.
* Expanded user, strategy, algorithm, and developer documentation.

The first 1,000 canonical puzzles are now a development regression suite. The
complete 49,158-puzzle corpus is the authoritative capability benchmark.

## Post-1.0 Priorities

The solver has reached functional completion. New work should improve quality,
performance, integration, and analysis without weakening the logic-first
contract.

### 1. Performance and Benchmark Engineering

* Profile strategy execution on the full corpus.
* Identify repeated candidate scans and avoid unnecessary allocation.
* Preserve strategy ordering and deduction behavior while optimizing.
* Add aggregate reporting across all canonical benchmark files.
* Track elapsed time, strategy usage, deductions, cells, and eliminations.
* Make performance comparisons reproducible across releases.

### 2. Output and Renderer Expansion

* Add CSV and TSV result output.
* Add explicit one-line puzzle and solution renderers.
* Add event-aware text and JSON renderers.
* Add compact replay and solution-path formats.
* Add optional color as a final presentation layer.
* Investigate Markdown, HTML, SVG, and GUI-oriented renderers.
* Keep renderer contracts versioned and solver behavior independent of output.

### 3. Puzzle Input, Canonicalization, and Identity

* Normalize supported blank markers through one input path.
* Improve validation and error diagnostics.
* Add canonical 81-character serialization.
* Add symmetry canonicalization where useful.
* Add stable puzzle fingerprints and duplicate detection.
* Support candidate-state interchange imports after the format contract is set.

### 4. Difficulty and Analysis

* Calibrate difficulty against a larger and more varied puzzle collection.
* Distinguish required strategy difficulty from incidental strategy use.
* Add strategy timing and search-effort metrics where meaningful.
* Preserve rating-version metadata so historical results remain interpretable.
* Investigate puzzle classification and comparative solve reports.

### 5. Test and Fixture Infrastructure

* Expand reusable puzzle fixtures and golden-output helpers.
* Add renderer snapshot and event-stream contract tests.
* Add temporary-file and CLI integration-test helpers.
* Retain the first-1,000 corpus as a regular regression target.
* Add practical full-corpus regression procedures for release verification.

### 6. Packaging and Developer Tooling

* Add continuous-integration coverage.
* Add distribution and tarball validation.
* Add release helpers and documentation-consistency checks.
* Improve installation and troubleshooting documentation.
* Keep `make check`, `git diff --check`, and benchmark verification central to
  release preparation.

### 7. User Interfaces and Generation

Longer-term possibilities:

* Puzzle generation with uniqueness verification.
* Difficulty-targeted generation.
* Interactive terminal interface.
* Desktop GUI or web interface.
* Teaching mode driven by deductions and renderer events.

## Strategy Policy After 1.0

Additional strategies are not required merely to increase the strategy count.
A new strategy should be considered when it:

* solves a demonstrated puzzle class more directly;
* produces a clearer proof than an existing fallback;
* materially improves performance or difficulty analysis; or
* supplies reusable inference infrastructure.

Each added strategy should include focused positive and negative tests,
structured deductions, human-readable explanations, documentation, benchmark
accounting, and a justified place in the registry.

## Long-Term Architecture

```text
Cell
    Represents one Sudoku cell and its candidate state.

Grid
    Represents puzzle state and provides lookup and mutation services.

Solver
    Controls strategy order, applies deductions, and records events.

Strategy
    Finds one or more logical deductions without mutating the live grid.

Deduction
    Represents a placement or candidate elimination and its reasoning.

Statistics
    Records solving metrics and strategy contributions.

Difficulty
    Rates a solve using a versioned interpretation of statistics.

Hint / Explain
    Presents the next deduction or a human-readable proof.

Benchmark
    Measures stable puzzle collections and detects regressions.

Renderer
    Converts grids, results, candidates, and events into output formats.
```

## Development and Release Guidelines

Every commit should ideally:

* Perform one logical task.
* Compile successfully.
* Pass `make check`.
* Avoid trailing whitespace.
* Include documentation updates when appropriate.
* Leave the repository in a releasable state.

Recommended release verification:

```bash
make PERL=perl check
git diff --check
make PERL=perl benchmark
```

For capability releases, also run the complete canonical corpus benchmark and
record the aggregate result.

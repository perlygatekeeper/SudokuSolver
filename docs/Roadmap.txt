# SudokuSolver Development Roadmap

## Project Philosophy

SudokuSolver is a logic-first solver and analysis tool. Its primary goal is to
solve puzzles through techniques that a human can inspect and understand.

Development should continue to follow these rules:

* Prefer logical deduction over unrestricted brute-force search.
* Keep solving strategies independent and directly testable.
* Require every deduction to explain why it is valid.
* Preserve backward compatibility whenever practical.
* Favor readable, maintainable code over clever code.
* Keep commits small and focused.
* Use tests and stable benchmarks to prevent regressions.

Bounded hypothetical inference is acceptable when it produces a finite,
human-verifiable proof. Recursive guessing and completed speculative solves
should not silently become the basis for ordinary deductions.

## Version 1.2.0 — Canonical Corpus and Reproducible Puzzle Generation

Status: complete and tagged

Version 1.2.0 turns the complete canonical 17-clue collection into a
queryable, reproducible puzzle corpus and generation platform. Work is divided
into nine phases.

### Phase 1 — Puzzle Representation

Goal: produce a deterministic, human-readable representation derived from an
81-character normalized puzzle string.

Deliverables:

* Preserve the normalized 81-character puzzle string as the source of truth.
* Add digit-grouped coordinate encoding in fixed digit order 1 through 9.
* Store row/column coordinate pairs without repeating their digit.
* Separate the nine digit groups with exactly eight delimiters.
* Produce exactly 42 characters for every 17-clue puzzle.
* Add clue-count and clue-location helpers.
* Validate generated encodings and audit all 49,158 corpus puzzles.
* Specify the encoding format in developer documentation.

Phase 1 is intentionally encode-only. It does not parse or decode arbitrary
coordinate encodings. Decoding is introduced when the definitive corpus gains
a genuine requirement to reconstruct puzzles from stored records.

Definition of done:

* All 49,158 corpus puzzles encode successfully.
* Every corpus puzzle has exactly 17 clues.
* Every generated encoding has nine groups and the expected length.
* Every generated encoding is unique within the source corpus.
* The encoding format is pinned by focused regression tests.

### Phase 2 — Sudoku Symmetry Model

Goal: represent every allowed Sudoku-preserving permutation explicitly and
apply it deterministically to normalized puzzle strings.

Development increments:

* [x] Represent and validate digit, row-within-band, column-within-stack, band,
  and stack permutations.
* [x] Apply a transform to a normalized puzzle string.
* [x] Serialize a transform into stable human-readable shorthand.
* [x] Invert transforms and verify exact restoration.
* [x] Compose transforms and verify equivalence to sequential application.
* [x] Parse serialized transforms for replay.
* [x] Generate deterministic random transforms from a supplied seed.

Status: complete

Definition of done:

* Every transform component is a validated permutation.
* Application preserves puzzle length and clue count.
* Applying a transform to a puzzle and its solution preserves consistency.
* Inversion restores the exact original puzzle.
* Composition matches sequential application.
* Serialization and parsing round-trip exactly.
* Reusing a random seed produces the same transform.

### Phase 3 — Canonization and Stable Identity

Development increments:

* [x] Add deterministic digit normalization by row-major first appearance.
* [x] Record the exact invertible digit transform in a canonical-result object.
* [x] Minimize row permutations within bands and band permutations.
* [x] Minimize column permutations within stacks and stack permutations.
* [x] Combine spatial and digit normalization into one deterministic representative.
* [x] Require idempotent full canonization and invariance under every supported symmetry.
* [x] Produce stable canonical fingerprints from canonical coordinate encodings.
* [x] Add a deterministic, parallel canonical staging-index generator.
* [x] Assign permanent corpus IDs from verified canonical ordering in Phase 4.

The first increment established the digit-normal form. The second increment
adds exhaustive row-side normalization across all 1,296 band and within-band
row permutations, digit-normalizing every candidate before lexical comparison.
The third increment adds the corresponding 1,296-candidate column-side normal
form across stack and within-stack column permutations. The fourth increment combines both spatial families and digit normalization in
an exhaustive correctness baseline spanning 1,679,616 spatial candidates. It
returns an exact witness transform and establishes idempotence and full symmetry
invariance. This baseline is not yet fast enough for the complete corpus;
the first-row pruning pass removes every transform pair that cannot match the globally minimal first row before full candidate construction. A second exact pruning stage evaluates only the 18 distinct leading-band row arrangements and selects the globally minimal first 27 characters. A third exact pruning stage evaluates the 216 distinct ordered two-band arrangements and selects the globally minimal first 54 characters before complete-grid construction. This removes the sixfold duplication caused by the final band row order. A dedicated canonicalization benchmark target tracks corpus-scale performance.
A deterministic canonical staging-index generator now records each source puzzle,
its canonical representative, 42-character fingerprint, and replayable witness
transform. Output order is stable across worker counts, and duplicate fingerprints
are rejected before Phase 4 assigns permanent corpus IDs.

The canonical fingerprint is now defined as the Phase 1 digit-grouped coordinate
encoding of the fully canonical 81-character puzzle. It is content-derived,
42 characters for every 17-clue puzzle, and invariant under all supported
symmetry transforms. Sequential `17C-NNNNNN` corpus IDs remain deferred until
canonical ordering is generated and verified in Phase 4.

### Phase 4 — Definitive Master Corpus

Development increments:

* [x] Assign stable `17C-NNNNNN` IDs by canonical puzzle ordering.
* [x] Add complete solutions.
* [x] Define and publish the authoritative JSONL master-corpus schema.
* [x] Add versioned difficulty scores and labels to the JSONL records.
* [x] Add highest required strategies to the JSONL records.
* [x] Add human-identifiable clue-pattern symmetries to the JSONL records.
* [x] Publish derived TSV and human-readable views.

Build one authoritative record for each of the 49,158 canonical 17-clue
puzzles. Each record includes:

* canonical ID and coordinate encoding;
* canonical 81-character puzzle and complete solution;
* versioned difficulty score and label;
* highest required strategy; and
* human-identifiable clue-pattern rotation and reflection symmetries.

The authoritative logical artifact is the master JSON Lines corpus, with one
canonical record per line. A development checkout may use either
`Puzzles/Master/sudoku17-master.jsonl` or the smaller
`Puzzles/Master/sudoku17-master.jsonl.gz`; the corpus reader accepts both. The
canonical index, identity, and solution TSV files are inspectable intermediate
build artifacts, not public corpus formats. Human-readable and tabular views
are derived from the JSONL master.

`make canonical-index` processes the complete input corpus by default. A
partial development build must be requested explicitly with `LIMIT=N`.

The solution-enrichment increment reads the permanent identity index, solves each
canonical puzzle in quiet mode, requires all 81 cells to be filled without a
contradiction, verifies every original clue against the solved grid, and writes
a deterministic solution-enriched TSV. The master-corpus builder then enriches
the JSONL records with versioned difficulty metadata, highest strategy, and
clue-mask symmetries without changing canonical identity or solution records.
Derived TSV and human-readable summary views are generated from the JSONL
master.

### Phase 5 — Corpus Query API

Status: complete

* [x] Provide lookup by canonical ID and coordinate fingerprint.
* [x] Expose a composable `select(...)` interface accepting multiple criteria at
  once with AND semantics.
* [x] Support difficulty, score, highest strategy, clue count, symmetry, ID, and
  fingerprint criteria.
* [x] Support ranges, multiple accepted values, exclusions, sorting, limits, and
  deterministic random selection.
* [x] Add convenience helpers that delegate to the common selection engine.

The public query API is implemented by `Sudoku::Corpus` and
`Sudoku::Corpus::Query`. It reads the authoritative JSONL master corpus,
indexes records by canonical ID and fingerprint, and returns chainable query
results for sorting, limiting, and deterministic random selection.

### Phase 6 — Symmetry-Randomized Puzzle Creation

Status: complete

* [x] Select a canonical corpus seed.
* [x] Apply a deterministic random symmetry transform.
* [x] Preserve both transformed puzzle and transformed solution.
* [x] Record the symmetry seed and explicit transform shorthand.

`Sudoku::Generator` implements symmetry-randomized creation on top of
`Sudoku::Corpus` and `Sudoku::Symmetry`. It accepts a corpus seed, a symmetry
seed, and optional corpus query criteria; returns a `Sudoku::GeneratedPuzzle`;
and records canonical ID, fingerprint, seeds, and transform shorthand for
future replay.

### Phase 7 — Controlled Clue Reveals

Status: complete

* [x] Reveal values from the transformed solution without removing original clues.
* [x] Reach an exact requested clue count.
* [x] Use a separate reveal seed.
* [x] Store the explicit reveal-cell list for future-proof replay.

`Sudoku::Generator->controlled_reveals(...)` builds on the Phase 6 symmetry
pipeline. It accepts the corpus seed, symmetry seed, reveal seed, and requested
final clue count; preserves the transformed pre-reveal puzzle as
`base_puzzle`; and records explicit `RrCc` reveal cells so replay can reconstruct
the final puzzle even if seeded random internals change later.

### Phase 8 — Difficulty-Targeted Generation

Status: complete

* [x] Generate within requested difficulty labels or score ranges.
* [x] Support highest-strategy and strategy-ceiling constraints.
* [x] Preserve the difficulty-rating version used to accept each puzzle.

`Sudoku::Generator->difficulty_targeted(...)` builds controlled-reveal
candidates, solves each final puzzle quietly, rates it with the versioned
`Sudoku::Difficulty` engine, and accepts only candidates matching requested
difficulty labels, score ranges, highest strategies, and strategy ceilings. The
accepted `Sudoku::GeneratedPuzzle` stores the rating version, label, score,
highest strategy, statistics snapshot, and number of generation attempts.

### Phase 9 — Provenance and Replay

Status: complete

Generated puzzle files remain readable in ordinary text editors and record:

* [x] generation date and generator version;
* [x] canonical ID and coordinate encoding;
* [x] symmetry seed and explicit transform;
* [x] reveal seed and explicit reveal list;
* [x] final clue count; and
* [x] versioned difficulty metadata.

Replay must reproduce the exact generated puzzle and verify its stored
provenance.

`Sudoku::GeneratedPuzzle->as_json` and `write_file(...)` produce readable JSON
artifacts. `Sudoku::Generator->replay(file => ...)` or `replay(data => ...)`
rebuilds the generated puzzle from the canonical corpus record, explicit
symmetry transform, and explicit reveal-cell list, then verifies stored puzzle,
solution, base puzzle, coordinate encoding, final clue count, and difficulty
metadata.

## Strategy Policy After 1.0

Additional strategies are not required merely to increase the strategy count.
A new strategy should be considered when it:

* solves a demonstrated puzzle class more directly;
* produces a clearer proof than an existing fallback;
* materially improves performance or difficulty analysis; or
* supplies reusable inference infrastructure.

Each added strategy should include focused positive and negative tests,
structured deductions, human-readable explanations, documentation, benchmark
accounting, and a justified place in the registry.

## Long-Term Architecture

```text
Cell
    Represents one Sudoku cell and its candidate state.

Grid
    Represents puzzle state and provides lookup and mutation services.

Solver
    Controls strategy order, applies deductions, and records events.

Strategy
    Finds one or more logical deductions without mutating the live grid.

Deduction
    Represents a placement or candidate elimination and its reasoning.

Statistics
    Records solving metrics and strategy contributions.

Difficulty
    Rates a solve using a versioned interpretation of statistics.

Hint / Explain
    Presents the next deduction or a human-readable proof.

Benchmark
    Measures stable puzzle collections and detects regressions.

Renderer
    Converts grids, results, candidates, and events into output formats.
```

## Development and Release Guidelines

Every commit should ideally:

* Perform one logical task.
* Compile successfully.
* Pass `make check`.
* Avoid trailing whitespace.
* Include documentation updates when appropriate.
* Leave the repository in a releasable state.

Recommended release verification:

```bash
make PERL=perl check
git diff --check
make PERL=perl benchmark
```

For capability releases, also run the complete canonical corpus benchmark and
record the aggregate result.
